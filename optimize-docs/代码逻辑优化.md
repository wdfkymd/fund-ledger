# fund-ledger 代码逻辑优化

审阅 `~/fund-ledger` 源码中的运行时逻辑、数据流、算法正确性。每项核实后再动手。

## 观察点

### 1. Finance 算法边界
`src/lib/finance.ts`（259行）是整个项目的金额/份额核心，审阅以下边界：

- **除法零除**：`calcProfitRate` 当 `costAmount===0` 时是否除零？应返回 `0` 或 `null`
- **null nav**：`calcMarketValue(shares, null)` 返回 `0` ✅（已验证有 null 守卫）
- **负份额/负成本**：发生 SELL 超卖或手工编辑导致负份额时，`calcProfit` 是否仍合理？不阻止但需返回值正确
- **多笔交易成本累加**：买入多笔后平均成本算法是否正确？从 `Holding.costAmount` 字段看是总成本法（Total Cost Average），不是平均成本法——确认 UI 端理解一致
- ✅ 核实后补充 JSDoc，写清"不再考虑已卖出份额的成本"的假设

### 2. 天天基金 API 零缓存（性能）
- `fetchFundFromEastMoney` / `fetchFundNavHistoryFromEastMoney` 每次直接打 eastmoney
- 高频路径：dashboard 刷新、基金详情页、批量刷新持仓时
- ✅ 建议：在 `fund-api.ts` 包一层内存 TTL cache（LRU，60s 有效期）
- ✅ 或挂 Next.js `unstable_cache`（但注意它是按 `fetch` URL 缓存，API 用了时间戳 bust）
- **如果当前无缓存且天天基金没限速，标记风险文档即可**，不必过度设计

### 3. Transaction 冲销逻辑
`PATCH /api/transactions/:id` 的 readme 描述"先冲销旧记录再应用新值"：

- ✅ 核实冲销计数是否精确（份额回滚 + 成本回滚）
- ❌ 多次 PATCH 同一笔交易是否会导致成本漂移？
- ✅ 验证 `transaction-ledger` 模式是否确保了幂等性

### 4. 并发写入 SQLite
SQLite + better-sqlite3 是同步单连接，不处理并发：

- ✅ 对个人项目足够了，但确认所有 db 写操作在 `withApi` 里串行
- ❌ 如果以后计划多用户并发写入大量刷新，当前模式会排队阻塞
- ✅ 文档化 SQLite 并发上限，留待日后决定是否迁移

### 5. N+1 查询风险
- `getHoldingsPayload` 看代码是 `prisma.holding.findMany({ include: { fund: true } })` 单次查询 ✅
- ✅ 但搜索基金/刷新净值等批量操作是否逐只查询 `fetchFundFromEastMoney`？
- ✅ `fetchFundNavHistoryFromEastMoney` 在基金详情页是单只拉取 ✅
- ✅ 批量刷新（`POST /api/funds`）要遍历用户所有持仓+自选逐只拉 eastmoney——这是串行还是并行？核实是否 `Promise.all`，如果不是改成并行

### 6. 进程内 rate-limit 缺陷
- `rate-limit.ts` 用模块级 `Map`，数据不跨实例共享
- ✅ 在注释里写明"单进程实例有效，多 worker 或多副本部署时失效"
- ✅ auth 限流若加了，也注明重启后会重置计数

### 7. JWT 会话与 Cookie
- `auth.ts` 用 jose 的 `SignJWT` / `jwtVerify`，7 天有效期
- ✅ secret 来自 `AUTH_SECRET` 环境变量，未设置时 `getSecret()` 抛错——合理
- ✅ 核实 logout 是否清除 cookie（`src/app/api/auth/logout/route.ts`）

### 8. 数据目录与文件清理
- `data/avatars/` 存上传头像，是否有机制清理未关联用户头像？
- ✅ 评估影响：个人项目头像数量有限，非必须

### 9. market-index.ts 指数行情
- 东财延迟行情，HTTP 调用
- ✅ 核实是否有缓存（单次请求？dashboard 加载即触发？）
- ✅ reading code：`src/app/\(dashboard\)/layout.tsx` 中 Server Component 直接获取一次传递给 client，client 在 `useEffect` 也拉一次——可能有冗余请求

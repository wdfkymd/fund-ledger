# fund-ledger 代码质量优化

审阅 `~/fund-ledger` 源码，按以下逐项核实 + 修复。半路接管项目，窄修窄验，不推倒重来。

---

## 观察点

### 1. Middleware 文件位置
- `src/proxy.ts` 导出了 `proxy()` 函数和 `config.matcher`，格式符合 Next.js middleware
- 但 Next.js 要求文件名必须是 `src/middleware.ts`
- ✅ 核实：`src/proxy.ts` 是否被手动导入（next.config / layout），还是纯死代码？
- ❌ 如果是死代码：重命名为 `src/middleware.ts`，或在实际 middleware 里导入它

### 2. `dev-docs/` 缺失
- AGENTS.md 将 `dev-docs/` 列为"内部真源"
- 但目录不存在（`.gitignore` 排除了它）
- ✅ 补一个最小 `dev-docs/HANDBACK.md`，包含：项目接管记录、遗留决策、已知技术债
- ✅ 或确认确实无记录后，修改 AGENTS.md 去掉 dev-docs 引用

### 3. 无测试
- package.json 无 test script，无测试依赖
- 核心算法 `src/lib/finance.ts`（259行，成本/收益/份额计算）**最该有测试**
- ✅ 给 finance.ts 写 vitest 测试，至少覆盖：
  - `calcMarketValue` / `calcEstimateValue`
  - `calcProfit` / `calcProfitRate` / `calcDayProfit`
  - 边界：null nav、负份额、零成本、大额精度
- ✅ 可选：验证 `npm run typecheck` 作为 CI 前置

### 4. `src/lib/actions/` 空目录
- 目录存在但为空
- ✅ 删除，或放一个 `.gitkeep` + 注释说明用途

### 5. 未捕获异常
- `fund-api.ts` 某些路径抛普通 `Error`，某些抛 `AppError`
- API 路由里有的 catch 了，有的未 catch（会导致 500）
- ✅ 统一：`fund-api.ts` 只抛 `AppError`（已定义在 `api.ts`）
- ✅ `withApi`  wrapper 要兜底未捕获异常，返回结构化的 500，不暴露内部细节

### 6. Auth 端点无限流（安全）
- `POST /api/auth/login`、`POST /api/auth/register` 无暴力破解防护
- 已有的 `rateLimit()` + `clientKey()` 未被 auth 路由使用
- ✅ 给两个端点加每 IP 限流：5 次/30s 窗口，超限返回 429

### 7. ESLint 配置较基础
- 仅 `eslint-config-next/core-web-vitals` + `typescript`
- ✅ 补 `eslint-plugin-perfectionist`（import 排序统一），或维持现状但 `npm run lint` 必须零报错

### 8. CSV 与文件安全
- `csv.ts` 已有公式注入防护（BOM + `'` 前缀）✅
- ✅ 验证 `export` API 只导出当前用户数据，无越权
- ✅ 验证 `avatar-fs.ts` 路径拼接防 `../`  目录遍历

### 9. Prisma 生成输出
- `prisma.schema` 指定 output=`../src/generated/prisma`
- `.gitignore` 已排除该目录 ✅
- ✅ 验证 `npm run build` 能正常生成，无 missing client 错误
